# 多数据源行情后台规划

本文记录当前仓库扩展到 Yahoo + 通达信 + Tushare 的拆分方案、数据库设计原则和已确认的环境事实，供后续逐子任务推进时对齐口径。

## 当前环境结论

2026-05-29 在本机按仓库默认配置直接探测 PostgreSQL，结论如下：

- PostgreSQL 实例可访问。
- 原默认业务库名 `strategy_studio` 不存在，因此服务按旧默认值无法连到业务库。
- 当前本机实际存在的业务库为 `etf_strategy`。
- `etf_strategy` 已存在平台核心表，但 Alembic 版本停留在 `20260525_0003`，落后于当前代码。

因此第一阶段必须先把“数据库连通性 + 业务库存在性 + 迁移是否到头”做成显式检查，并把默认配置切回当前实际业务库。

## 目标范围

最终后台需要同时支持：

- Yahoo：抽取约 100 个全球活跃代码，持续管理 `1m`、`15m`、`1d` 三类 K 线。
- 通达信：从本地 `vipdoc` 导入全部可见市场与周期的原始 K 线。
- Tushare：保存公司行动事件，作为通达信前复权重算输入。
- 前复权：把公司行动先转换为公式区间，再批量重算前复权数据。
- 任务化：前端能按渠道发起下载/导入任务，后台自动入库并记录进度、失败原因和增量状态。

## 子任务拆分

### 子任务 1：数据库底座与诊断

- 新增 `check-db` 命令与平台 API，输出业务库存在性、迁移版本和表概览。
- 新增多数据源基础表，先把渠道、代码映射、统一 K 线序列、公司行动、前复权公式区间、文件增量状态和任务状态落库。

### 子任务 2：统一数据源服务层

- 设计统一的渠道注册、序列查询和任务创建接口。
- 前端市场数据页改造成“按渠道管理”的后台入口，而不是只支持 Yahoo 单标的补数。
- 兼容旧 `price_bars` 读取逻辑，逐步切换到新 `market_data_series + market_data_bars`。

### 子任务 3：Yahoo 批量样本入库

- 固化 100 个全球活跃标的样本池及其默认市场口径。
- 批量拉取 `1m / 15m / 1d`，写入统一序列表。
- 为免费分钟线窗口补齐增量策略、失败重试和覆盖统计。

### 子任务 4：通达信原始数据导入

- 迁移 `F:\trade\tdx\python\csv_parser` 的核心解析逻辑，但不修改原仓库。
- 先接通达信原始数据导入和文件级 manifest。
- 把市场、周期、源路径和源代码映射写入统一序列表。

### 子任务 5：Tushare 公司行动与前复权公式

- 迁移 `tdx` 的 Tushare 公司行动抓取逻辑到当前仓库。

- 参考 `corp_actions_cpp` 的区间 A/B 公式设计，生成 `price_adjustment_segments`。

### 子任务 6：前复权批量重算与任务编排

- 参考 C++ 双指针线性匹配思路，避免 Python 版逐行扫描事件。
- 先生成公式区间，再批量重建前复权日 K。
- 把原始导入、公司行动抓取、公式生成、前复权重算串成统一后台任务。

## 新数据库结构

当前已落地的多源基础表如下：

- `data_providers`：渠道注册表，记录 Yahoo、通达信、Tushare。
- `instrument_aliases`：同一标的在不同渠道中的代码映射，例如 `600519.SS` 对应 `sh600519`。
- `market_data_series`：统一定义“某渠道、某代码、某周期、某复权口径、某 session”的一组 K 线。
- `market_data_bars`：统一 K 线事实表。
- `corporate_action_events`：公司行动事件表。
- `price_adjustment_segments`：前复权公式区间表，保存 `A/B` 仿射参数。
- `source_file_manifests`：通达信文件增量状态表。
- `data_ingestion_jobs` / `data_ingestion_job_items`：统一后台任务与子项进度表。

旧表 `instruments / price_bars / data_sync_runs` 仍保留，用于平滑迁移当前 Yahoo 单通道流程。后续会逐步把读写主路径切换到新结构。

## 设计取舍

- 公司行动、公式区间、原始 K 线、复权 K 线分表管理，不把“原始事实”和“派生结果”混在同一行里。
- 通达信文件增量状态独立建表，而不是把文件 hash 混进任务表；这样任务失败和文件状态可以分别恢复。
- 前复权采用“先公式、后重算”，为后续迁移 C++ 算法留出稳定接口。
- 统一任务表按“渠道 + 范围 + 选项 + 子项明细”建模，后续前端不需要为 Yahoo、通达信、Tushare 分别造三套后台任务 UI。
