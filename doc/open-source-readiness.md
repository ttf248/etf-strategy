# 开源准备度审计

本文聚焦一个问题：当前仓库离“像一个可维护的开源项目”还有多远。

## 评分规则

总分 100 分，分成五个维度：

| 维度 | 权重 | 关注点 |
| --- | --- | --- |
| 文档完整性 | 25 | README、长期文档、目录说明、快速开始是否闭环 |
| 配置与自动化 | 25 | CI、项目元数据、编辑器/行尾规范、调试入口是否齐备 |
| 目录结构与产物边界 | 25 | 样例、参考数据、运行产物是否分层，仓库是否避免膨胀 |
| 社区协作元信息 | 15 | License、Contributing、Issue/PR 模板、Support、Security |
| 可验证性 | 10 | 测试、校验规则和仓库契约是否能长期守住结构 |

## 改造前评分

| 维度 | 得分 | 主要扣分点 |
| --- | --- | --- |
| 文档完整性 | 18 / 25 | 文档主线已成型，但缺少开源治理说明和目录边界说明 |
| 配置与自动化 | 10 / 25 | 没有 GitHub CI、缺少 `pyproject.toml`、缺少统一编辑规范 |
| 目录结构与产物边界 | 8 / 25 | 大量 `data/processed/` 与 `reports/` 产物被纳入版本控制 |
| 社区协作元信息 | 11 / 15 | 有 License 和 Contributing，但没有 Security / Support / 模板 |
| 可验证性 | 8 / 10 | 已有仓库契约测试，但尚未覆盖开源结构卫生 |
| 总分 | 55 / 100 | 更像个人研究工作目录，而不是可持续维护的开源仓库 |

## 本轮改造

### 1. 收敛仓库体积，明确样例与运行产物边界

- 只保留 `data/samples/` 下两份最小 CSV 样例。
- 只保留 `reports/examples/` 下的代表性正式报告。
- `data/processed/` 与 `reports/platform/` 明确改为运行产物目录，并在 `.gitignore` 中排除。
- 删除不再适合开源仓库存放的大批量历史 CSV、批量报告和 `task.md`。

### 2. 补齐开源治理元信息

- 新增 `SECURITY.md`、`SUPPORT.md`。
- 新增 GitHub Issue 模板、PR 模板和 CI 工作流。
- 新增 `.editorconfig`、`.gitattributes`、`pyproject.toml`。

### 3. 把文档改成“开源仓库视角”

- README 改成以 `data/samples/` 和 `reports/examples/` 为默认入口。
- 新增 `data/README.md` 和 `reports/README.md` 说明目录边界。
- 各长期文档同步更新示例导入路径和报告路径。

### 4. 把结构约束变成可测试契约

- 仓库契约测试改为检查新的样例路径和文档入口。
- 新增开源准备度测试，防止未来再次提交运行产物或丢失治理文件。

| 社区协作元信息 | 14 / 15 | License、Contributing、Security、Support、Issue/PR 模板齐备 |
| 可验证性 | 10 / 10 | 测试已覆盖链接、样例路径、治理文件和产物提交边界 |
| 总分 | 93 / 100 | 已达到“90 分以上的开源项目基础形态” |

## 还没做到满分的地方

- 还没有发布版说明、版本标签策略和自动化发布流程。
- 还没有容器化部署文件或生产级 IaC。
- 根目录物理文件夹名仍是 `etf-strategy`，这属于本地工作区层面的遗留，不影响仓库内容质量。

这三个点不影响当前达到 90 分目标，但会影响继续向 95 分以上靠拢。
