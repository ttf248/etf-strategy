# Codex 项目规则

本文件分为两层：

- **通用规则**：适用于本仓库内所有 AI 协作、文档维护和代码修改任务。
- **项目专属规则**：只针对本 ETF 策略回测项目的入口、依赖、数据链路、报告和调试配置。

## 通用规则

### 语言、验证与提交

- 文档、报告和面向用户的命令行输出默认使用中文。
- Git 提交日志必须使用中文，简洁描述本次变更目的。
- 每个独立任务完成后必须运行相关验证，并用中文提交。
- 每个提交只包含当前任务相关文件，不混入本地输出目录、缓存文件或无关修改。

### 任务记录

- AI 执行编码或文档维护任务时，必须同步更新 `task.md`。
- `task.md` 需要记录本轮任务状态、修改方案、修改内容和设计取舍，方便后续继续迭代。

### 任务拆分

- 如果检测到任务规模较大，必须先按可验证、可提交的粒度拆分子任务。
- 每完成一个子任务，就运行对应验证并提交 Git，然后继续下一个子任务。
- 拆分子任务时优先按清晰的功能边界划分，避免一个提交同时混入多个不相关目标。

### 注释与可维护性

- 整个项目代码需要保持“适度注释”标准。
- 新增或修改代码时，应补充足够注释，优先解释业务规则、边界条件、失败处理和设计取舍，避免出现大段难以维护的无注释逻辑。
- 代码注释默认使用中文，只解释关键业务规则、二进制格式、异常处理和不可从代码直接看出的设计取舍，不复述显而易见的代码流程。
- 新增或修改复杂模块时，应补充模块级说明、关键函数 docstring 或关键分支注释，确保后续维护者能理解数据口径、边界条件和失败处理策略。

## 项目专属规则

### 运行入口与依赖

- Python 程序以根目录 `main.py` 的 `main()` 函数作为启动入口。
- 不使用 `python -m 包名` 作为主要运行方式。
- 不在代码中硬编码 `.venv` 路径；VS Code Python 管理器负责选择解释器。
- 新增第三方库时必须同步维护 `requirements.txt`。

### ETF 策略工程边界

- 子任务拆分应优先贴合本项目的功能边界，例如数据扫描、CSV 解析、公司行动抓取、前复权计算、MySQL 导入、绘图和文档测试分别处理。
- 涉及行情数据、参数寻优、回测结果、图表或 Markdown 报告的变更时，必须明确说明数据口径、默认样本、输出路径和验证方式。
- 本地运行生成的临时输出、缓存、日志和一次性实验产物不得随功能提交混入版本控制，除非它们已经被明确约定为默认正式样本或正式报告产物。

### 命令、文档与调试配置同步

- 新增命令或重要参数时，必须同步更新 README 和测试。
- 新增或修改常用运行入口、默认参数、默认样本、报告产物时，必须同步更新相关文档和 `.vscode/launch.json`。
- 只要代码改动影响用户使用方式、运行入口、默认输出或报告阅读路径，就必须同步更新 README、`doc/` 文档和 VS Code 调试配置。
